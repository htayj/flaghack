# NetHack feature gap analysis for FlagHack

Status snapshot last verified against FlagHack commit `2ba89d6` (`feat(gameplay): stream state and enrich creatures`).

Sources compared:

- NetHack 3.6 Guidebook, especially sections 2-11 (`doc/Guidebook.txt` in the local NetHack source checkout).
- NetHack 5.0.0 source at tag `NetHack-5.0.0_Released`, cloned under `~/reference/external_src/github.com/NetHack/NetHack`.
- FlagHack domain/server/client source schemas and actions.

This is a feature parity inventory, not a claim that FlagHack should copy every NetHack mechanic exactly. FlagHack already intentionally diverges with a Burning Man-style campground, theme camps, coolers, hippies, and an Effect HTTP plus revisioned SSE client-server architecture.

## High-level summary

FlagHack currently has the skeleton of a NetHack-like roguelike:

- ASCII/tile-style map rendering in the Charm terminal client.
- 8-direction movement, wait/noop, and run/travel conveniences in the terminal client.
- A status/message/inventory UI.
- A starting role-selection flow, currently with one role.
- A generated campground overworld plus BSP-style dungeon generation support.
- Walls, floors, tunnels/roads, tents, signs, effigy/temple terrain, and openable/closable doors.
- Basic creatures with six rolled attributes, centralized tag capability masks, and simple NPC movement.
- Basic item entities, inventory letters, pickup/drop, cooler containers, loot take/put, eat, and quaff.
- Save/restore/quit lifecycle with NetHack-like save-exit and restore-consumes-save semantics.
- Revisioned authoritative full-client-state streaming to the main clients, with stale-revision rejection and HTTP refresh fallback.

Compared with NetHack, most deep roguelike systems are still absent or only represented as placeholders. The largest gaps are combat/death, multi-level dungeon progression, rich object/equipment mechanics, monster ecology, traps/shops/religion/quests, identification/curses, and character progression.

## NetHack 5.0 source-backed reference points

The NetHack 5.0 source makes the gap larger and more concrete than the Guidebook alone. It exposes not just player-facing commands, but the data structures and subsystems behind them.

### Source scale and data model

- NetHack 5.0.0 is confirmed by `include/patchlevel.h` (`VERSION_MAJOR 5`, `VERSION_MINOR 0`, `PATCHLEVEL 0`).
- The 5.0 source tree inspected has roughly 132 `src/` files, 90 `include/` files, and 154 `dat/` files at the release tag.
- `include/monsters.h` contains 394 `MON(...)` monster entries, each with level, speed, AC, MR, alignment, generation flags, up to six attacks, size/nutrition/sound, resistances, monster flags, difficulty, and color. FlagHack currently has a handful of creature tags, required six-attribute records, and cheap per-tag capability masks (brain/eyes/hands/humanoid/Kop/egregore/mindless), but no comparable per-species combat/stat block.
- `include/objects.h` is a macro-driven object database covering generic object classes and hundreds of object entries across weapons, armor, rings, amulets, tools, food, potions, scrolls, spellbooks, wands, gems, rocks, balls/chains, venom, coins, and corpses. FlagHack currently models only flags, basic comestibles/drinks, coolers, hammer, and nails.
- `include/obj.h` shows that individual NetHack objects carry quantity, weight, object type/class, inventory letter, artifact index, location state, timers, BUC state, unpaid/no-charge shop state, known/discovery flags, erosion, locked/broken/trapped state, poison/grease/bypass flags, corpse species, partial-eaten nutrition, age, worn-slot mask, and more. FlagHack items currently have only entity key, tag, position, and container ownership.
- `include/you.h` shows the hero state includes location history, level/depth, movement energy, trap state, room/shop occupancy, hunger, timed properties, sickness, vision ranges, polymorph state, hit points while polymorphed, events, conduct, roleplay, attributes, alignment, luck, hit/damage bonuses, and many other fields. FlagHack's player has an optional role until setup completes and a required six-attribute record, but lacks the rest of this hero-state model.
- `include/rm.h`, `include/trap.h`, `include/dungeon.h`, `include/monst.h`, and `include/permonst.h` reveal separate rich models for map cells, traps, dungeon branches/special levels, live monsters, and monster species. FlagHack currently stores nearly everything as flat world entities in one `HashMap`.

### Additional gaps revealed by the 5.0 command table

The earlier Guidebook-based table already lists the major systems. Reading `src/cmd.c` adds several smaller but still player-visible missing commands/mechanics:

- `#dip`: dipping objects into potions/fountains/water.
- `#enhance`: weapon/spell skill advancement UI.
- `#force`: forcing locks.
- `#invoke`: artifact and special-object powers.
- `#jump`: jumping to another square.
- `#offer`: altar sacrifice.
- `#pray`: prayer.
- `#ride`: mount/dismount saddled steeds.
- `#rub`: rubbing lamps/stones.
- `#sit`: sitting effects.
- `#tip`: emptying containers.
- `#turn`: turn undead.
- `#untrap`: removing traps or helping trapped entities.
- `#wipe`: wiping face/cream/blindness-related state.
- `#terrain`: viewing map terrain without monsters/objects.
- `#overview`, `#chronicle`, `#conduct`, `#vanquished`, `#genocided`: meta/progress/history views.
- `#herecmdmenu` / `#therecmdmenu`: context-sensitive command menus.
- Wizard/debug-only command families for map/monster/object/time/vision inspection.

These features mostly depend on deeper systems FlagHack does not yet have: object effects, locks, traps, altars, undead, face/vision state, mounts, skills, artifacts, conduct tracking, and player knowledge/discovery databases.

## Current FlagHack features that are closest to NetHack

These are the strongest parity points today:

1. **Core roguelike presentation**: ASCII-style world, message log, status, inventory, and map drawing.
2. **Directional controls**: `hjkl/yubn` movement plus run/travel-like helpers in the terminal client.
3. **Doors**: closed doors block, open doors pass, movement into a closed door opens it, and explicit open/close actions exist.
4. **Inventory letters and item popups**: item-letter selection for pickup/drop/eat/quaff/loot flows.
5. **Containers**: coolers act as loot containers with take/put flows.
6. **Food/drink commands**: eat/quaff exist, although they only consume/remove items.
7. **Save lifecycle**: save exits and clears memory; explicit restore consumes save; quit deletes save.
8. **Generated spaces**: NetHack-like BSP rooms/corridors exist, alongside a FlagHack-specific large campground overworld.

## Biggest missing systems by gameplay impact

If the goal is to become recognizably NetHack-like, the highest-impact missing systems are:

1. **Combat and death**
   - HP, attacks, damage, AC, death, corpses, monster hostility, and combat messages.
   - Without this, creatures are mostly blockers or simple movers rather than threats.

2. **Dungeon progression**
   - Stairs/ladders, multiple persistent levels, dungeon depth, branches, special levels, and a goal path.
   - This turns the current map into an actual run structure.

3. **Equipment and item mechanics**
   - Weapons, armor, wield/wear/remove, item stacks, weights, BUC/enchantment/charges, and item effects.
   - This unlocks meaningful inventory decisions.

4. **Monster ecology**
   - Many monster types with stats, behavior, attacks, generation rules, difficulty scaling, and corpses.

5. **Hunger/time/status effects**
   - Nutrition, food effects, conditions, regeneration, speed, timed properties, and encumbrance.

6. **Traps/secrets/shops**
   - These provide much of NetHack's environmental texture and risk/reward.

7. **Identification and randomness**
   - Unknown object appearances, discovery, naming/calling, and blessed/cursed/cursed variants.

8. **Quest/win condition/scoring**
   - Amulet-equivalent objective, role quest, endgame, score, and conduct tracking.

## Suggested staged roadmap toward NetHack parity

### Stage 1: Make the game dangerous

- Add HP/death and simple melee combat.
- Add monster hostility and adjacent attacks.
- Add damage messages and a game-over terminal state.
- Add corpses or at least death/removal for monsters.

### Stage 2: Make a run progress

- Add stairs/ladders and multi-level state.
- Persist levels by `z` depth.
- Connect campground to dungeon entrance.
- Add a minimal objective item and win/escape condition.

### Stage 3: Make inventory matter

- Add item stacks/counts and weights.
- Add weapon/armor entities and equipment slots.
- Implement wield, wear, take off/remove, and basic AC/damage.
- Make food/drink affect hunger/HP/status rather than only disappear.

### Stage 4: Add NetHack texture

- Add traps and searching.
- Add locked/secret doors and kicking.
- Add shops/gold/shopkeepers.
- Add basic scrolls/potions/wands with unidentified appearances.

### Stage 5: Add deep NetHack systems

- Roles/races/gender/alignment/deities.
- Pets and tameness.
- Prayer, altars, sacrifice, conduct.
- Quest branch and role artifacts.
- Bones levels, scoring, explore/debug modes.

## Notes on intentional differences

Some NetHack gaps may remain intentional for FlagHack:

- The campground/burn setting is a strong distinct identity and should not be treated as a missing dungeon feature.
- Coolers, theme camps, hippies, rangers, acid cops, flags, effigies, and temples can map onto NetHack-like systems without copying NetHack's exact content.
- The Effect HTTP plus revisioned SSE architecture is very different from NetHack's monolithic local process; full-state stream revisions address stale snapshots, but lifecycle and stale-command handling still need to remain stricter because requests can overlap.
